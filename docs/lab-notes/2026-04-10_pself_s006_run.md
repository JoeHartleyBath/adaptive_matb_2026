## Post-hoc analysis — LORO failure investigation (2026-04-10)

Full investigation conducted after the session to understand why LORO Youden J = 0.0.
Diagnostic scripts created: `scripts/_tmp_loro_s006_diagnose.py`, `_tmp_c1_timing.py`, `_tmp_amplitude_check_s006.py`, `_tmp_block_level_features_s006.py`, `_tmp_loro_perrun_norm_s006.py`, `_tmp_loro_threshold_explain_s006.py`, `_tmp_10fold_threshold_s006.py`, `_tmp_adapt_perrun_s006.py`.

### Root cause chain (both confirmed empirically)

**Cause 1 — C1 label contamination (offset bug)**

- C1 XDF had 0 MATB markers. `_load_xdf_block` fell back to scenario-file timing with default `MATB_SCENARIO_OFFSET_S = 12.0 s`. Actual offset = **0.943 s** (measured from C2, which had markers). Error = **11.06 s per block**, contaminating ~19% of each block's windows with the wrong label.
- With corrected offset alone: fold_j = [0.008, 0.157] — still fails the 0.10 gate. Cause 2 dominates.

**Cause 2 — Electrode settling amplitude drift (dominant)**

- Raw EEG RMS: rest = 22,642 → cal_c1 = 46,319 → cal_c2 = 81,168 (4× increase over ~20 min). Electrode settling, not physiology.
- `norm_stats.json` derived from rest recording → Beta/Hjorth features are **15–19 Z-units too high in C2** and large-negative in C1 after Z-scoring.
- Combined effect: the C2-trained model assigns inverted probabilities to C1 features. Fold 0 AUC = 0.38 (below chance — systematic inversion). Pooled ROC degenerates → threshold = Infinity.
- 38/40 features shift significantly between C1 and C2 at all workload levels (Mann-Whitney U, BH-corrected), with monotonic within-C2 drift block-by-block.

### Fixes tested

| Fix | LORO fold J | Pooled J | Passes 0.10? |
|---|---|---|---|
| Original (offset=12 s, rest norm) | [0.0, 0.076] | 0.0 | No |
| Corrected offset only (0.943 s) | [0.008, 0.157] | 0.071 | No |
| Corrected offset + per-run LOW norm | [0.324, 0.165] | **0.228** | **Yes** |

Per-run LOW norm: normalise each XDF by its own LOW-block feature mean/std instead of the global rest baseline. FM_Beta global offset between C1 and C2 drops from +13.7 Z to −0.54 Z.

### Why the LORO pooled threshold is near-zero (0.033) despite J=0.228

The two LORO folds produce different absolute probability scales because each is a different model:

| Fold | Training data | HIGH mean p_high | Fold-optimal threshold |
|---|---|---|---|
| 0 (test=C1) | C2 | 0.43 | 0.368 |
| 1 (test=C2) | C1 | 0.12 | 0.029 |

Pooling forces the argmax down to 0.033 to capture fold 1's HIGH windows. This is a structural artefact of 2-fold pooling with mismatched probability scales — not a meaningful deployment value. Figure: `results/figures/s006_loro_threshold_explanation.png`.

### Better threshold: 10-fold stratified CV on combined C1+C2

With per-run LOW norm, 10-fold stratified CV on combined C1+C2 (same procedure as the reference paper):

| Method | Threshold | Pooled J | AUC |
|---|---|---|---|
| LORO pooled argmax | 0.033 | 0.228 | — |
| Training-set | 0.214 | 0.699 | 0.924 |
| **10-fold CV** | **0.366** | **0.679** | **0.911** |

Youden and min-distance-to-(0,1) criteria agree exactly (both 0.366). Every fold J > 0.63 — no degenerate folds.

Note: 10-fold J/AUC is optimistic because every fold mixes C1+C2 windows — no fold tests true cross-run transfer. LORO J=0.228 remains the honest cross-session number. The 10-fold threshold (0.366) is nonetheless a better deployment value than the LORO pooled threshold.

### Adaptation offline replay (per-run LOW norm, 10-fold threshold = 0.366)

Adaptation XDF normalised by its own LOW-block stats (block_01 recovered: 55 s available; block_06 fully labelled). Model fitted on combined C1+C2 per-run-normalised data.

| Metric | Value |
|---|---|
| Assist ON | 47% (7 activations, 6 deactivations) |
| LOW blocks mean p_high | ~0.10 |
| HIGH blocks mean p_high | 0.40–0.62 |
| MOD blocks mean p_high | ~0.39 (straddles threshold — appropriate) |
| C2 → adaptation drift (per-run norm) | mean \|Δ\| = 0.38 Z across all features+levels |

LOW drift = ~0 by construction (per-run LOW norm anchors each session to its own LOW level). HIGH drift is genuine workload structure change across sessions. Transfer is adequate.

Figures:
- `results/figures/adaptation_s006__perrun_norm_10fold_threshold.png`
- `results/figures/s006_loro_threshold_explanation.png`

### Summary of recommended pipeline changes for `calibrate_participant.py`

1. **Norm**: replace rest-baseline norm with per-run LOW-block norm inside `_compute_loro_threshold()`. The deployed full-fit model should also use LOW (not rest) norm for Z-scoring.
2. **Threshold selection**: replace LORO pooled argmax with 10-fold stratified CV on combined C1+C2. LORO J ≥ 0.10 becomes a quality gate only (warn but still use 10-fold threshold on failure).
3. **Marker warning**: `_load_xdf_block` must log a named warning when 0 MATB markers are detected, so the operator is alerted during calibration rather than failing silently.
