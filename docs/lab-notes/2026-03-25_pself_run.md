### Phase 2 — Staircase calibration

**Status:** [x] Issues — staircase did not converge; session stopped manually at ~16:30

**What happened:**
- Scenario `adaptation_skeleton.txt` launched with `--adaptation` flag ✓
- AdaptationScheduler initialised correctly ✓
- Tracking scores throughout: ~0.77–0.83 (consistently **above** target 0.70)
- Difficulty stayed at `d=1.000` the entire run — no step-downs ever fired
- Session stopped manually at ~112 s (scenario was 300 s); exit code 2 (early exit detection)

**Root cause: `d_init=1.0` in `AdaptationConfig`**
- The staircase started at **maximum difficulty** (`d=1.0`), already at the ceiling
- With score > target, the staircase tried to step *up* — but there was nowhere to go
- Since `d` was clamped at 1.0 on every STEP UP attempt, the staircase could never find the target and would never converge
- The correct starting point is `d_init=0.5` (midpoint): if performance is too high the staircase steps up to find threshold; if too low it steps down

**Fix required (tomorrow):**
- In `src/adaptation/adaptation_scheduler.py`, change `AdaptationConfig.d_init` from `1.0` to `0.5`

```python
# AdaptationConfig default:
d_init: float = 0.5   # start at midpoint so staircase can go up or down
```

**Additional note on exit code 2:**
- The early-exit error (code 2) was triggered because the scenario ran only ~112 s of 300 s (below the 90% threshold in `run_openmatb.py`)
- This was correct behaviour — it was triggered by the **manual stop**, not a real crash
- Once the staircase fix is applied and the scenario runs to full convergence, this error will not occur
